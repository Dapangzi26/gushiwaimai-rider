# [OPEN] rider-login-failure

## 现象
- 骑手端当前点击登录后无法正常进入工作台。
- 用户截图显示停留在 `pages/login/index`，暂未提供登录按钮点击后的控制台或 toast 反馈。

## 当前目标
- 确认登录失败发生在：
  - 提交前表单校验
  - 登录接口请求
  - 登录接口返回后的角色/审核校验
  - 登录成功后的本地存储或页面跳转
  - 近期调试埋点对登录流程造成副作用

## 初始假设
- H1：登录按钮事件已触发，但被前端表单校验提前拦截。
- H2：登录接口请求失败或返回非 2xx，页面未给出明显反馈。
- H3：登录接口成功，但返回账号角色不是 `rider`，被登录页逻辑拦住。
- H4：登录接口成功且角色正确，但 `setToken/setUserInfo/switchTab` 之后又被其他逻辑立刻清会话。
- H5：最近加入的调试埋点会在登录页 `onLoad/onShow` 或请求链路上产生副作用，导致无法完成登录。

## 证据状态
- 暂未收集本轮专用运行时日志。
- 需要补充登录提交、接口返回、角色判断、跳转结果的精确运行时证据。

## 证据更新
- 用户提供控制台截图出现明确报错：
  - `TypeError: fetch is not a function`
  - 报错位置覆盖 `App.vue onLaunch / onShow` 与 `pages/login/index.vue onLoad / onShow`
- 结论：当前“登录按钮点不动”不是登录接口故障，而是最近加入的调试埋点使用了 `fetch`，在 uni-app App 原生环境不可用，导致页面生命周期先崩溃。

## 处理决策
- 按最小修复原则，不改登录业务逻辑。
- 仅将调试上报从 `fetch` 替换为 `uni.request`，保留调试能力并恢复原生环境兼容性。
