# 站长定位超时调试记录

## 症状
- 乡镇站长登录后，控制台持续出现定位超时日志。
- 用户反馈“之前还是好的”，说明存在回归窗口。
- 当前希望先查清站长定位超时被哪个环节改坏，再处理乡镇骑手位置上报问题。

## 预期
- 站长账号能稳定获取定位并上报到后端。
- 调度台能够持续看到站长位置。
- 定位失败时应有明确证据，能区分是权限、系统定位、坐标系、网络上报还是触发条件问题。

## 初始假设
1. 我前面改动会话刷新或后台任务启动条件后，导致站长端定位上报触发频率或触发时机变化，放大了原本偶发的定位超时。
2. 当前设备/基座环境不支持 `gcj02`，代码进入 `wgs84` 兜底后仍因系统定位能力不足而持续超时，和业务逻辑本身无关。
3. 站长账号身份字段在登录后发生变化，导致 `canReportDispatchLocation()` / `startLocationReport()` 走了和之前不同的分支。
4. 真实定位已获取，但 `/rider/location/report` 上报失败或被限频，日志表象被误判成“定位超时”。
5. 之前关闭调试上报或补会话刷新时，影响了定位链路某处异步时序，导致 `App Show -> 会话校验 -> 定位启动` 的先后顺序变化。

## 待采集证据
- 站长账号登录后的身份字段：`role / delivery_scope / rider_kind / town_name`
- `canReportDispatchLocation()` 的判定结果
- `startLocationReport()` 是否真的启动、启动了几次
- `uni.getLocation()` 每次请求的参数与失败类型
- `/rider/location/report` 的成功 / 失败 / throttled 返回
